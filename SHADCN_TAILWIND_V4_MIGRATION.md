# Codex 执行规约：将 geshu-rsbuild-template 衍生项目迁移到 shadcn/ui + Tailwind CSS 4

## 任务定位

你是负责实际执行迁移的 Codex，而不是向开发者讲解概念的文档助手。

收到本文件后，应在当前目标项目中完成迁移、验证结果并汇报。不要只输出操作建议，也不要要求用户手动复制配置，除非目标项目存在必须由用户决定的兼容性冲突。

迁移目标：

- 从 Ant Design + Tailwind CSS 3 迁移到 shadcn/ui + Tailwind CSS 4。
- 使用 shadcn 预设 `b6HGg41FS`。
- shadcn 初始化参数必须包含 `--pointer true`，即 `pointer` 的目标值固定为 `true`。
- 最终必须启用可点击按钮的 `pointer` 光标行为。
- 保留目标项目原有业务能力、交互语义、校验逻辑和项目约定。
- 后续可以继续使用 shadcn CLI 添加组件，并让生成组件使用目标项目配置的别名。

除上述目标外，所有可能随时间、CLI 版本、依赖版本、项目结构或预设内容变化的信息，都必须在执行时实时获取。禁止从本文件、模型记忆或其他旧项目复制生成后的 CSS Token、依赖版本、组件源码和浏览器版本。

## 一、必须遵守的执行原则

### 1. 先读取目标项目规则

开始修改前，必须：

1. 查找并完整读取目标项目适用范围内的 `AGENTS.md`。
2. 检查 Git 状态，识别用户已有改动。
3. 读取 `package.json`、包管理器配置、构建配置、TypeScript 配置、样式入口、应用入口和全局 Provider。
4. 确认目标项目是否仍然基于 Rsbuild，以及目录是否与模板一致。

目标项目的 `AGENTS.md`、用户当前指令和现有代码风格优先于本文件中的通用建议。

不得覆盖、回滚或格式化与本次迁移无关的用户改动。

### 2. 必须实时获取易变信息

执行时必须通过当前 CLI 和官方文档确认以下信息：

- shadcn CLI 当前支持的命令、参数和框架。
- 预设 `b6HGg41FS` 当前解析出的完整内容。
- 该预设当前生成的主题 CSS、字体、组件风格和直接依赖。
- Tailwind CSS 4 当前的安装方式和浏览器最低要求。
- Rsbuild 当前推荐的 Tailwind CSS 4 集成方式。
- shadcn 当前推荐的通知、表单和基础组件实现。

只使用官方一手资料：

- <https://ui.shadcn.com/docs>
- <https://tailwindcss.com/docs>
- <https://rsbuild.rs/guide>

技术文档和 CLI 行为可能已经变化。即使本文件中的示例命令仍能运行，也必须先查看当前帮助信息。

### 3. 不得写死生成结果

禁止直接从本文件或其他已迁移项目复制以下内容：

- `index.css` 中的预设 Token。
- `components/ui` 下的组件源码。
- 预设对应的字体、颜色、圆角数值和暗色主题细节。
- 精确依赖版本。
- Tailwind CSS 4 的浏览器最低版本。
- shadcn CLI 的可用参数列表。

这些内容必须由执行时的最新版 CLI、目标项目依赖解析结果和官方文档产生。

可以固定的只有用户指定的目标，例如预设代码 `b6HGg41FS`、迁移到 Tailwind CSS 4、`--pointer true`，以及目标项目已有的业务约束。

### 4. 不得伪装框架

如果 shadcn CLI 不能识别 Rsbuild：

- 不要在目标项目中强行指定 Vite、Next 或其他错误模板。
- 不要为了通过检测而临时伪造框架文件。
- 应创建一个可安全删除的临时参考项目，让当前 CLI 生成该预设的最新结果，然后把与框架无关的生成结果适配到目标项目。

临时项目只能用于获取当前事实，不是最终交付物。读取完成后，应验证绝对路径并安全清理。

## 二、执行流程

### 阶段 1：目标项目预检

至少检查以下内容：

```powershell
git status --short
git branch --show-current
Get-Content package.json -Raw
Get-Content tsconfig.json -Raw
Get-Content rsbuild.config.ts -Raw
```

继续定位：

- Tailwind CSS 入口和旧配置文件。
- PostCSS 配置及其中除 Tailwind 外的其他插件。
- Ant Design 的 Provider、全局消息、主题、表单和组件引用。
- `dayjs`、字体包、图标库是否被业务独立使用。
- `@/*` 别名实际指向根目录还是 `src`。
- 是否已经存在 `components.json`、`components/ui` 或 shadcn 工具文件。
- ESLint 和 Prettier 的配置形式。
- 项目原有的类型检查、Lint、测试和构建命令。

使用 `rg` 搜索实际引用，不要只根据 `package.json` 推断：

```powershell
rg -n 'antd|@ant-design|soda-antd|message\.|ConfigProvider|StyleProvider|FormSubscribe' -g '!node_modules'
rg -n 'tailwind|postcss|autoprefixer|dayjs|source-han-sans' -g '!node_modules' -g '!pnpm-lock.yaml'
```

如果工作区已有改动，先判断哪些属于用户正在进行的迁移，哪些与本任务无关。在已有迁移改动基础上继续，不要重置整个工作区。

### 阶段 2：实时检查 shadcn CLI 和预设

先检查当前 CLI：

```powershell
npx --yes shadcn@latest --help
npx --yes shadcn@latest init --help
npx --yes shadcn@latest add --help
npx --yes shadcn@latest preset --help
```

解析目标预设：

```powershell
npx --yes shadcn@latest preset decode b6HGg41FS
```

将实际输出作为本次迁移的来源，不要假设它与以前的解析结果相同。

用户指定的固定输入是 `--pointer true`，不得省略或改为 `false`。同时检查 `pointer` 在当前 CLI 中是带值参数、布尔开关还是已经改名：如果当前 CLI 接受显式值，则传入 `--pointer true`；如果当前 CLI 将它声明为布尔开关，则使用语义等价的 `--pointer`；如果参数已经变更，则使用当前官方等价方式实现 `pointer=true`。

### 阶段 3：获取当前预设的准确生成结果

优先使用 CLI 原生能力，但必须先判断是否会安全修改目标项目。

#### 情况 A：当前 CLI 已正式支持目标 Rsbuild 项目

1. 确认工作区可回滚。
2. 使用当前 CLI 帮助中声明的语法传入预设和 `pointer=true`。
3. 让 CLI 初始化或应用预设。
4. 立即检查 Git 差异，确认没有改写业务文件或采用错误目录。

#### 情况 B：当前 CLI 仍不能识别 Rsbuild

在系统临时目录中生成参考项目：

1. 使用 CLI 当前支持的、与本项目最接近的 React 客户端模板。
2. 使用预设 `b6HGg41FS`，并按当前 CLI 语法传入 `pointer=true`。
3. 不启动参考项目。
4. 从参考项目读取：
    - `components.json`；
    - 全局 CSS；
    - 包清单；
    - 工具函数；
    - 基础组件生成结果；
    - 构建集成所需信息。
5. 区分“预设生成内容”和“参考框架专属内容”。只将前者适配到目标项目。
6. 验证临时目录绝对路径确实位于预期临时根目录后再递归删除。

临时项目的生成命令也必须根据当前 `init --help` 组合，不要在本文中固定模板参数和选项顺序。

### 阶段 4：接入 Tailwind CSS 4

打开当前 Rsbuild 官方 Tailwind CSS 4 文档，采用执行时官方推荐的集成方式。

迁移时必须：

1. 更新 Tailwind CSS 到当前兼容的 4.x 版本。
2. 安装当前 Rsbuild 官方方案所需的插件。
3. 在 Rsbuild 配置中注册插件。
4. 确保全局 CSS 仍由应用入口导入。
5. 将旧 Tailwind 配置中的自定义主题、插件、扫描路径和 safelist 迁移到 Tailwind CSS 4 对应机制。
6. 确认旧 PostCSS 配置是否还承担其他职责，再决定修改还是删除。

不得因为模板默认的 `tailwind.config` 很简单，就假设所有衍生项目也可以直接删除该文件。

如果项目使用动态拼接的 Tailwind 类名，应改成可以静态扫描的完整类名。真正运行时才知道的尺寸、坐标、颜色或资源地址可以使用 `style` 或 CSS 变量。

### 阶段 5：写入当前预设，而不是历史快照

目标项目的主题配置必须来自阶段 3 的当前生成结果。

处理要求：

- 保留目标项目仍然需要的全局业务样式。
- 删除已经失效的 Ant Design 专用样式和层级声明，但只能在对应组件迁移后删除。
- 不要把参考项目的入口路径、框架插件或目录结构原样复制到 Rsbuild 项目。
- 如果目标项目必须保留原中文字体，应在不破坏预设语义的前提下适配字体回退，并在最终报告中说明偏差。
- 如果目标项目已有暗色主题策略，合并当前预设 Token，不要重复创建相互竞争的主题系统。

### 阶段 6：配置 shadcn CLI 的本地路径

根据目标项目真实结构创建或更新 `components.json`。

必须保证：

- Tailwind CSS 4 下的配置字段符合当前 shadcn schema。
- CSS 路径指向目标项目真实入口。
- `components`、`ui`、`hooks`、`lib`、`utils` 别名与 `tsconfig` 一致。
- 工具别名指向目标项目约定的 shadcn 工具文件。
- 后续 `shadcn add` 生成的导入可以被 TypeScript 和 Rsbuild 同时解析。

若目标项目仍采用本模板的根目录结构，优先沿用以下目录约定：

- CLI 生成组件：`components/ui`。
- shadcn 工具文件：`utils/shadcn.ts`。
- 公共业务组件：`components`，不要放入 `components/ui`。

不要手写 `components/ui` 的组件实现。创建好 `components.json` 后，根据目标项目实际使用到的 Ant Design 组件，通过当前 shadcn CLI 添加对应组件。

先使用 CLI 当前提供的预览或差异能力检查写入内容，再执行添加。不要一次添加所有组件。

### 阶段 7：迁移 Ant Design，允许阶段性共存

不要先删除 Ant Design 依赖。

按以下顺序迁移：

1. 全局主题和 Provider。
2. 全局消息与通知。
3. 公共封装组件。
4. 简单页面组件。
5. 表单、弹窗、选择器和复杂表格。
6. 最后清理依赖与样式。

迁移每个组件时，按语义寻找当前 shadcn 推荐方案，不要做机械的同名替换。

必须保留：

- 表单校验规则。
- 提交中、成功和失败状态。
- 禁用条件。
- 查询缓存刷新。
- 键盘操作和可访问性。
- 弹窗关闭、确认和取消语义。
- 表格分页、筛选、排序和选择状态。
- 原有响应式布局和桌面交互。

简单表单可以使用 React 19 的 Form Action；复杂表单应根据当前 shadcn 官方方案和项目已有依赖选择合适表单库。不要用大量零散状态替代原本完整的表单模型。

全局消息迁移时，使用当前 shadcn 官方推荐组件。不要保留伪造的全局 Ant Design `message` 兼容层。应修改真实调用方，使类型和运行时语义一致。

如果目标项目包含大量 Ant Design 页面，可以先让两套组件共存。每完成一批迁移就执行类型检查和 Lint，直到旧引用清零。

### 阶段 8：安装和清理依赖

依赖处理必须以当前生成结果和实际导入为准：

- 先让 shadcn CLI 安装它声明的依赖。
- 再通过类型检查发现遗漏的直接依赖。
- 对照临时参考项目的包清单核对预设所需运行时包。
- 不要从本文复制固定版本。
- 遵守目标项目的 pnpm 供应链策略、最小发布时间和构建脚本白名单。

只有在 `rg` 确认无引用后，才能删除：

- Ant Design 及其样式运行时。
- `soda-antd`。
- 仅服务于旧 Tailwind CSS 3 的依赖。
- 仅服务于 Ant Design 的日期或字体依赖。

如果 `dayjs`、字体、图标或 PostCSS 插件被业务独立使用，必须保留。

删除依赖后再次搜索旧导入，不要以包管理器成功为完成标准。

### 阶段 9：更新浏览器目标

执行时打开 Tailwind CSS 官方兼容性文档，读取当前 Tailwind CSS 4 的最低浏览器要求。

然后：

1. 读取目标项目当前 `.browserslistrc` 或等价配置。
2. 与产品实际需要支持的浏览器范围比较。
3. 如果产品目标允许，更新到不低于 Tailwind CSS 4 当前官方要求的范围。
4. 如果产品必须支持低于官方要求的浏览器，不要擅自放弃兼容性。应将其作为迁移阻塞项报告给用户，并说明需要继续使用 Tailwind CSS 3 或调整产品支持范围。

禁止把某次历史文档中的浏览器版本直接写入所有项目。

### 阶段 10：处理 ESLint 和 Prettier

先检查目标项目当前配置格式，再合并忽略规则。

对于本模板家族，默认忽略：

- `components/ui`；
- shadcn 工具文件，例如 `utils/shadcn.ts`。

这样可以避免项目格式化规则反复改写 CLI 生成源码。

要求：

- 不要覆盖目标项目已有忽略项。
- 确认 ESLint 配置导出是数组还是对象，再选择合并方式。
- 用 Prettier 的文件信息命令验证文件确实被忽略。
- 用 ESLint 的显式文件检查验证匹配了忽略规则。

如果目标项目规则明确要求检查生成组件，则以目标项目规则为准，不要强行忽略。

## 三、必须执行的验证

### 1. 静态验证

至少执行目标项目对应的：

```powershell
pnpm exec tsc --noEmit
pnpm lint
pnpm build
git diff --check
git status --short
```

如果项目有测试命令、组件测试或端到端测试，应根据变更风险执行。

不得启动开发服务，除非用户明确允许或目标项目规则要求。

### 2. 旧技术栈残留检查

使用 `rg` 搜索：

- Ant Design 包和子路径导入。
- `soda-antd`。
- 全局 `message`。
- `.ant-*` 样式。
- Tailwind CSS 3 指令。
- 已删除字体和日期依赖的引用。
- 旧 PostCSS 插件配置。

对每个残留结果分类：

- 必须迁移；
- 迁移期暂时保留；
- 业务独立使用，应永久保留；
- 文档或历史文本，无运行时影响。

不要为了让搜索结果为空而删除仍有业务价值的依赖。

### 3. shadcn 配置验证

使用当前 CLI 提供的项目预设解析命令，确认目标项目最终解析为 `b6HGg41FS`。

如果当前 CLI 已经更改或移除该命令：

1. 重新查看 CLI 帮助。
2. 使用当前官方替代命令。
3. 如果不存在替代命令，将目标项目配置与本次实时生成的参考项目逐项比较。

同时验证新添加的组件确实从目标项目配置的工具别名导入，而不是引用错误的 `@/lib/utils`、`src` 路径或参考项目路径。

### 4. 运行时与视觉验证

只有在用户授权启动项目时才进行运行时验证。

若允许验证：

- 普通 Web 项目使用项目合适的本地测试方式。
- Electron 项目不要在普通浏览器中加载，应遵守目标项目的 Electron 验证规则。
- 检查亮色和暗色主题。
- 检查 Toast、表单、禁用状态、响应式布局和键盘交互。
- 检查字体、圆角、主题色和焦点环是否符合本次实时解析出的预设。

## 四、故障处理规则

### 1. CLI 执行失败

如果 CLI 失败：

1. 检查它已经写入了哪些文件和依赖。
2. 保存完整错误信息。
3. 区分框架检测失败、网络失败、包策略失败和配置错误。
4. 不要在失败结果上连续叠加不明补丁。
5. 复盘本轮改动，撤回没有依据且属于本轮的改动，再采用最小替代方案。

不得回滚用户在任务开始前已有的工作区改动。

### 2. 生成组件缺少依赖

以 TypeScript 的实际模块缺失错误和生成组件的导入为依据补装直接依赖。不要凭旧项目的 `package.json` 批量复制依赖。

补装后重新执行类型检查、Lint 和构建。

### 3. 预设解析结果不一致

如果最终无法解析为 `b6HGg41FS`：

1. 重新解码目标预设。
2. 对比本次临时参考项目和目标项目的配置。
3. 检查样式、基础色、主题、字体、圆角、图标库、菜单配置和 CSS 路径。
4. 修正根因，不要只改预设解析输出所依赖的某一个表面值。

### 4. 旧浏览器要求冲突

这是产品兼容性决策，不是普通代码错误。不要自行降低浏览器支持范围。停止对应迁移步骤，向用户报告 Tailwind CSS 4 与现有最低浏览器要求的冲突，并等待选择。

## 五、完成标准

只有同时满足以下条件，才能报告迁移完成：

- 目标项目规则已被遵守。
- 用户原有无关改动未被覆盖。
- Tailwind CSS 4 已按当前 Rsbuild 官方方案接入。
- 主题来自本次实时生成的 `b6HGg41FS`，不是历史 CSS 快照。
- 固定输入 `--pointer true` 已按当前 CLI 语义传入，或通过官方等价方式实现。
- `pointer` 行为已正确启用。
- shadcn CLI 可以继续在目标项目中添加组件。
- 生成组件的目录和工具导入别名正确。
- Ant Design 业务能力已经迁移，或所有阶段性保留项已明确报告。
- 不再需要的旧依赖和样式已经安全清理。
- 浏览器目标已经与当前 Tailwind CSS 4 要求和产品要求协调。
- ESLint、Prettier、TypeScript、Lint 和生产构建验证通过。
- 临时参考目录已经安全清理。
- Git 差异只包含本次迁移范围。

## 六、最终汇报格式

最终回复应简洁说明：

1. 实际采用的 shadcn CLI、预设解析结果和 Tailwind/Rsbuild 集成方式。
2. 修改了哪些基础设施、Provider、业务组件和消息系统。
3. 删除了哪些旧依赖，保留了哪些仍有业务用途的依赖。
4. 浏览器最低要求是否与产品要求兼容。
5. 执行了哪些验证及结果。
6. 是否存在未迁移模块、阶段性共存项或需要用户决定的兼容性问题。

如果用户同时要求提交，则在所有验证通过后按目标项目的 Git 提交规范创建提交；否则不要擅自提交或推送。
